Monitoring & Observability

Moving past basic uptime checks. Learn how to transform raw distributed system telemetry data into actionable operational insights using modern metrics frameworks and open-source telemetry pipelines.

Monitoring vs. Observability: The Fundamental Paradigm Shift

Many engineering organizations use these terms interchangeably, but they represent entirely different operational approaches:

  • Monitoring: Tracks known failure modes ("Is my CPU usage over 85%?", "Is the application answering on port 443?"). It tells you when a pre-defined threshold breaks.
  • Observability: Infers the internal states of a system based solely on its external outputs (Telemetry). It allows you to debug novel, unexpected edge failures ("Why are users in a specific region experiencing 500 errors only when checking out with a specific payment method?").

The Core SRE Telemetry Frameworks

To build reliable dashboards that do not overwhelm on-call engineers with alarm fatigue, SREs standardize around structured observability frameworks.

1. Google SRE's 4 Golden Signals

Optimized for general distributed user-facing software architectures:

Signal Definition Practical Target Example
Latency The time it takes to service a request. p99 Latency < 250ms
Traffic A measure of how much demand is being placed on the system. HTTP requests/sec (RPS)
Errors The rate of requests that fail explicitly or implicitly. HTTP 5xx responses > 0.01%
Saturation How "full" your system is (memory, disk I/O constraints). AWS ECS Memory Utilization

2. The RED Method

Specifically tailored for modern Microservice environments and API endpoints:

  • Rate: The volume of requests per second your services are processing.
  • Errors: The absolute number or percentage of those requests that fail.
  • Duration: The precise distribution profile of time those requests take to complete.

🛠️ Enterprise Tooling Playbook

Throughout my 27-year operational career at enterprises like The Hartford, Scratch Foundation, and LexisNexis, I've managed complex monitoring platform migrations. Here is how you map those exact telemetry platforms to these frameworks:

  • Datadog: Excellent for unified application performance monitoring (APM), serverless AWS Lambda tracing, synthetic end-to-end testing hooks, and calculating native SLO tracking bars directly from custom infrastructure telemetry logs.
  • New Relic & Splunk: Leverage Splunk's powerful text processing search queries to isolate distributed application error signatures alongside New Relic APM agent instrumentation lines to maintain end-to-end p95/p99 transaction insight profiles.
  • Sumo Logic, CA Spectrum, & CA UIM: Critical telemetry nodes for central log aggregation, synthetic fault infrastructure tracking, network fabric monitoring, and processing baseline operational alert distributions.

OpenTelemetry Collector Configuration Blueprint

To avoid vendor lock-in, modern SRE architectures leverage the vendor-agnostic OpenTelemetry (OTel) Collector pipeline. Below is a complete configuration template used to collect, process, and stream metrics/traces out to Prometheus and Jaeger endpoints. 

# otel-collector-config.yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318
  prometheus:
    config:
      scrape_configs:
        - job_name: 'otel-collector'
          scrape_interval: 15s
          static_configs:
            - targets: ['localhost:8888']

processors:
  batch:
    timeout: 1s
    send_batch_size: 256

exporters:
  prometheus:
    endpoint: "0.0.0.0:8889"
    namespace: "sreconcepts"
  otlp/jaeger:
    endpoint: "jaeger:4317"
    tls:
      insecure: true

service:
  pipelines:
    metrics:
      receivers: [otlp, prometheus]
      processors: [batch]
      exporters: [prometheus]
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlp/jaeger]
