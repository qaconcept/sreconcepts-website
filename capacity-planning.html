Capacity Planning

A Senior SRE Guide to Proactive Resource Management

“Capacity planning is the art of having exactly enough resources – not too much (waste), not too little (outage).”

Part 1: Core Principles of SRE Capacity Planning

1.1 What is Capacity Planning?

Capacity Planning is the proactive practice of predicting, allocating, and scaling computing resources (CPU, memory, network, storage) to guarantee applications run reliably while balancing infrastructure footprint efficiency and overall cloud cost.

1.2 The Capacity Planning Golden Equation

To determine resource thresholds safely, platform engineers leverage a strict mathematically backed logic profile:

Required Capacity = (Peak Load × Safety Factor) + Redundancy
Variable / Input Parameter Architectural Description Typical Industry Standard Values
Peak Load Maximum expected concurrent requests per second (RPS) profile. Derived via predictive time-series forecasting metrics.
Safety Factor Structural calculation buffer added to account for structural market uncertainty. 1.2 – 1.5 (yielding a 20% to 50% head-room margin).
Redundancy Extra unallocated compute layers preserved for immediate hard failover mitigation. 1 to 2 completely isolated extra task instances (N+1, N+2).

1.3 Why Capacity Planning Fails (Common Pitfalls)

Identified Pitfall Downstream Consequence SRE Defensive Prevention Pattern
Reactive scaling only Cascading outages caused during sudden traffic spike steps before clusters provision. Implement proactive trend forecasting tied to baseline auto-scaling parameters.
Overprovisioning “just in case” Massive capital deficit wasting approximately 40%–60% of absolute cloud budget. Automated resource right-sizing matched to tightly monitored utilization targets.
Ignoring system dependencies The downstream database layers lock up, becoming a severe systemic barrier. Execute end-to-end load testing to expose systemic limits.
Single-region architecture designs A solitary regional cloud provider outage triggers complete target system failure. Enforce cross-regional multi-zone distributed active-standby or active-active capacity footprints.
Static metrics thresholds Applying fixed rules (e.g., “80% CPU”) fails as microservice runtimes evolve yearly. Establish automated continuous profiling and rolling infrastructure evaluations.

3.3 Simple Forecasting Formula (Linear Growth Profile)

Forecast_RPS = Current_RPS × (1 + Growth_Rate)^(Days/30)

Practical Example Scenario: A cluster handles a base of 10,000 RPS, running standard 5% month-over-month growth patterns, evaluated exactly 90 days out into the upcoming production window:

10,000 × (1 + 0.05)^(90/30) = 10,000 × (1.05)^3 = 11,576 RPS Target Peak

Standard Enterprise Forecasting Engine Frameworks:

  • Prometheus Layer: Leveraging the native automated predict_linear() structural time-series functions.
  • Amazon Web Services: Deploying managed AWS Auto Scaling predictive resource profiling.
  • Datadog Platforms: Using modern integrated ML-backed Datadog Forecast Alerts.
  • Custom Frameworks: Direct mathematical implementations inside Excel matrices, Python numerical processing tasks, or specialized R code libraries.

Part 4: Load & Stress Testing – Finding Your Limits

4.1 Types of Capacity Tests

Load Test Archetype Primary Systemic Testing Goal Target Volume Curve Range Test Execution Duration
Smoke test Validation testing ensuring configurations run correctly without breaking. 1% – 5% of overall standard target volume footprint. 1 Minute Baseline
Load test Empirical mapping of standard platform performance configurations. Exactly 100% of validated forecasted cluster peak volumes. 30 – 60 Minutes
Stress test Exposing system breaking points and component dependency thresholds. 120% – 200% of maximum target peak. Maintained continuously up to system failure
Soak test Isolating creeping application performance degradation and lingering memory leak factors. 80% of current standard maximum runtime load. 4 – 24 Structural Hours
Spike test Profiling downstream runtime scaling speed and resource burst capacity limits. Instantaneous surge scaling from 0 to 200% capacity. 5 Minute Window

4.2 Capacity Per Task (Single Request/Unit calculations)

Capacity Per Task = (Total Requests Handled) / (Peak Request Rate × Time)

For practical service deployment engineering scenarios, use this simplified latency relationship logic:

Max RPS per instance = 1 / (Average response time in seconds)

Numerical Application Profile: If an atomic microservice tracking indicates an average overall execution latency profile of 50 milliseconds (0.05 seconds):

Max RPS per instance = 1 / 0.05 = 20 RPS Per Microservice Container Instance

Once isolated, the infrastructure engineer extracts the overarching cluster allocation count requirements directly via:

Instances needed = (Forecast peak RPS / Max RPS per instance) × (1 + Safety Buffer)

Substituting standard verification numbers: Target peak traffic load is 15,000 RPS, verified hardware limit handles 20 RPS per unit, maintaining a clear 20% safety margin buffer (1.2 multiplier):

Required Instances = (15,000 / 20) × 1.2 = 750 × 1.2 = 900 Total Live Nodes

5.2 Calculating N+1 for Microservices Configurations

If runtime metrics indicate a target cluster requires exactly 10 container units to comfortably process maximum production surges:

  • N+1 Model Allocation: Deploy 11 total instances. A single node can drop offline during hardware failure with zero user degradation.
  • N+2 Model Allocation: Deploy 12 total instances. Preserves capacity for an entire node to fail during concurrent system patches or rolling adjustments.

Part 6: Key Capacity Metrics to Monitor

6.1 The Four Golden Signals for Capacity Management

Golden Signal Domain Capacity Operational Definition Direct Infrastructure Planning Implication Standard Production Alerting Thresholds
Saturation An empirical tracking of exactly how full a specific computing unit is. Critical indicator pointing to imminent system exhaustion or outage risk. CPU > 80%, Memory > 85%, Disk Volume Space > 80%
Utilization The percentage tracking of active efficiency for provisioned infrastructure. Exposes idle resources and opportunities for cost optimization. CPU < 20% maintained continuously for more than 24 hours
Traffic The overarching metric tracking incoming user request rates. Signals broad, long-term application usage growth trends. Month-over-month usage increases scaling past 20%
Errors The transactional failure tracking rate across systems. Exposes service errors caused directly by hardware resource limits. Aggregated HTTP 5xx responses scaling above 1% during peaks

Part 7: Modern Implementation Strategies

7.1 Auto-scaling Archetypes (Horizontal vs Vertical Dimensions)

Auto-scaling Type Underlying Infrastructure Mechanism Best Suited Application Architecture Common Production System Examples
Horizontal Scaling (Out/In) Dynamically adjusts the total node instance count. Stateless application layers and decoupled task microservices. Kubernetes HPA, KEDA Event Schedulers
Vertical Scaling (Up/Down) Alters the designated resource allocations of individual nodes. Stateful legacy databases and tightly coupled single-node backends. Cloud Instance Size Adjustments, Kubernetes VPA
Predictive Auto-scaling Applies ML models to prepare infrastructure ahead of forecast spikes. Highly cyclical environments with noticeable latency step spikes. AWS Predictive Scaling Engines

Standard Cool-Down Configuration Policy (Anti-Flapping Thresholds)

Scaling Execution Direction Minimum Enforced Cool-down Window Maximum Configured Cooldown Multiplier
Scale-Out Phase (Adding Resources) 1 Minute Baseline Buffer 5 Minute Maximum Guardrails
Scale-In Phase (De-provisioning nodes) 5 Minute Cooldown Window 15 Minute Enforced Cooldown Safety

7.2 Capacity Spike Readiness Runbook Blueprint

Runbook Profile: Anticipated Traffic Spike Incident Actions

Pre-Event (T-24h Verification): Force configuration scales manually to 1.2x estimated maximums. Lock and freeze scale-in automation. Warm edge caching layers.
During Event Operations: Evaluate system dashboard tracking every 5 minutes. If compute saturation metrics exceed 85% continuously for 120 seconds, execute a 50% node scale-out step.
Post-Event De-provisioning: Gradually scale back nodes to baseline, monitoring for lingering traffic tails. Restore default auto-scaling behaviors.

7.3 Chaos Engineering Validation Matrices

Simulated Chaos Experiment Underlying Technical Fault Emulation Method Expected Automated Systemic SRE Outcome
Sudden Node Instance Failure Hard terminate exactly 20% of active running live containers. Remaining production node pools absorb the load with zero request drops (N+1 validation).
Network Latency Insertion Inject a fixed 500ms delay block across 50% of application request routes. Downstream telemetry triggers automated platform auto-scaling rules.
Complete Zone Blackout Route all active user connections completely away from an isolated Availability Zone. The alternative live infrastructure pools handle traffic shifts without breaching SLO budgets.
Database Replica Overload Execute heavy analytical parsing loops continuously against production replicas. The primary transactional database tier remains isolated with zero downstream degradation.
Extreme Traffic Surges Inject a 2x user request load burst within a tight 10-second window. The auto-scaling group scales out within 120 seconds to stabilize performance metrics.
